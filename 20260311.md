## 1. 경쟁사 벤치마킹 요약

### 1.1 RankBreeze — 검색 순위 직접 조작

| # | 벤치마킹 서비스 | 핵심 로직 | 벤치마킹 이유 |
|---|----------------|----------|-------------|
| R1 | 편의시설 분석기 | 비교군 전체 편의시설 빈도 카운트 → 누락 항목 식별 | 구현 난이도 낮고 즉시 가치 제공 |
| R2 | 키워드 허브 | 상위 리스팅 제목/설명에서 n-그램 추출 → 빈도 순위 | 제목 최적화의 기반 데이터 |
| R3 | 경쟁자 사진 비교 | 자사 vs 경쟁자 사진 수량/커버사진 나란히 비교 | 시각적으로 즉각 이해 가능 |
| R4 | 검색 순위 추적 | 게스트 수/날짜별 일일 순위 기록 | 최적화 효과 측정의 핵심 지표 |

**수집 데이터**: 일일 검색 순위, 경쟁 리스팅 데이터 (가격/사진/편의시설/점유율), 지역 시장 키워드
**고유 기능**: SEO 캠페인 (실제 사용자를 보내 리스팅 클릭 → CTR 직접 부스트)

### 1.2 AutoRank — AI 자동 최적화

| # | 벤치마킹 서비스 | 핵심 로직 | 벤치마킹 이유 |
|---|----------------|----------|-------------|
| A1 | AI 제목 생성 | 시장 키워드 + 랭킹 기준 + 현재 제목 → 최적화 제목 | 가장 빠른 ROI |
| A2 | AI 설명문 재작성 | 셀링 포인트 + 리뷰 테마 + 키워드 → 설명문 | 전환율 직접 영향 |
| A3 | 리뷰 퓨전 | 전체 리뷰 → 상위 속성 추출 → 리스팅 반영 | 게스트 언어로 셀링 포인트 발굴 |
| A4 | 주간 자동 업데이트 | 트렌드 변화 감지 → 리스팅 자동 갱신 | 지속적 최적화 |

**수집 데이터**: 수백만 리스팅의 데이터, 시장별 상위 검색 키워드, 전체 게스트 리뷰
**고유 기능**: 7일마다 자동 업데이트 + 컨시어지 서비스 (전문가 + AI 조합)

### 1.3 AirDNA — 시장 데이터 분석

| # | 벤치마킹 서비스 | 핵심 로직 | 벤치마킹 이유 |
|---|----------------|----------|-------------|
| D1 | 비교군 자동 매칭 | 유형/규모/위치 기반 유사 리스팅 탐색 | 모든 분석의 기반 인프라 |
| D2 | RevPAR/ADR 벤치마킹 | 비교군 대비 재무 성과 위치 파악 | 호스트가 가장 관심 갖는 수치 |
| D3 | 점유율 트렌드 | 12개월 점유율/수익 트렌드 차트 | 장기 성과 추적 |
| D4 | 공급/수요 지표 | 시장 전체의 리스팅 증감/수요 변화 | 시장 포화도 판단 |

**수집 데이터**: RevPAR, ADR, 점유율, 예약 리드타임, 공급/수요 지표
**고유 기능**: 10개 비교 시장 대비 월간 성과 비교, 12개월 트렌드 차트

### 1.4 PriceLabs — 동적 가격 엔진

| # | 벤치마킹 서비스 | 핵심 로직 | 벤치마킹 이유 |
|---|----------------|----------|-------------|
| P1 | 비교군 가격 추적 | 350개 인근 리스팅 일일 가격 수집 | 가격 포지셔닝의 기반 |
| P2 | 이벤트 기반 가격 | 지역 이벤트/공휴일 → 가격 인상률 계산 | 이미 부분 구현됨 |
| P3 | 요일/계절 패턴 | 요일별/월별 수요 패턴 학습 → 가격 반영 | 기본적이지만 효과 큼 |
| P4 | 예약 속도 감지 | 비교군 예약 속도 → 수요 급증 감지 | 실시간 가격 대응 |

**핵심 알고리즘 (HLP)**: 15km 반경 350개 유사 리스팅 기반, 매일 가격 재계산
**입력 신호**: 예약 속도, 계절성, 지역 이벤트, 경쟁자 요금, 날씨, 항공편 데이터
**고유 기능**: 150개+ PMS 연동, 미래 지향적 (과거 데이터가 아닌 향후 수요 기반)

### 1.5 경쟁사 비교표

| 관점 | RankBreeze | AutoRank | AirDNA | PriceLabs |
|------|-----------|----------|--------|-----------|
| 주요 목표 | 검색 순위/가시성 | 자동 최적화 | 재무 분석 | 수익 관리 |
| 데이터 초점 | 경쟁자 (10K) | 검색 트렌드 (수백만) | 시장 재무 | 동적 신호 |
| 핵심 강점 | 사진/키워드 벤치마킹 | AI 기반 지속 업데이트 | 재무 벤치마킹 | 일일 가격 최적화 |
| 업데이트 주기 | 수동 + 캠페인 | 주간 자동 | 월간/분기 | 일일 |
| 비교군 규모 | 10,000+ | 시장 전체 | 10개 비교 시장 | 350개 초지역 |

### 1.6 StayTrend 고유 강점 (경쟁사에 없는 것)

- **국적별 마케팅 채널 추천**: GA4 국적 데이터 → 국가별 최적 채널 매핑
- **Instagram/GA4 통합 분석**: 숙소 성과를 SNS 마케팅과 연결
- **외국인 비율 기반 가격 조정**: 외국인 게스트 비율에 따른 가격 승수 적용
- **올인원 포털**: 전략/가격/콘텐츠/분석/리포트 단일 대시보드 통합

## 2. StayTrend 통합 파이프라인 설계

### 2.1 5단계 파이프라인 구조

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  STAGE 0: 비교군 엔진 (Foundation)
  ─────────────────────────────────
  입력: 자사 리스팅 좌표, 숙소 유형, 수용 인원, 침실 수
  처리: 동적 반경 내 유사 리스팅 50~100개 탐색 (스크래핑)
  출력: comp_set (비교군 리스팅 목록 + 전체 필드)
  저장: properties/{id}/comp-set/{listingId}
  주기: 온보딩 시 1회 + 월 1회 갱신

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  STAGE 1: 시장 인텔리전스 (Market Intelligence)
  ──────────────────────────────────────────────
  comp_set 데이터 기반 4개 분석 병렬 실행:
    [P1] 가격 벤치마킹 → 날짜별 중앙값, 자사 백분위
    [D2] 수익 벤치마킹 → RevPAR, ADR 비교
    [R1] 편의시설 격차 분석 → 누락 항목, 희귀 고효과 편의시설
    [R2] 키워드 분석 → 상위 성과자 제목/설명 n-그램
  저장: properties/{id}/market-intel/{category}
  주기: 월 1회 (비교군 갱신 시)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  STAGE 2: 리스팅 진단 (Listing Diagnosis)
  ────────────────────────────────────────
  자사 리스팅을 비교군 대비 점수화:
    [R3] 사진 진단 → 수량, 커버 사진, 품질
    [A3] 리뷰 인사이트 → 감성, 토픽, 카테고리별 트렌드
    콘텐츠 진단 → 제목 길이/키워드, 설명 토픽 커버리지
    운영 진단 → 응답률, 즉시 예약, 캘린더, 취소율
  출력: 종합 Health Score (0~100)
  저장: properties/{id}/diagnosis/{category}
  주기: 온보딩 시 + 월 1회

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  STAGE 3: AI 최적화 제안 (AI Recommendations)
  ────────────────────────────────────────────
  Stage 1 (시장 데이터) + Stage 2 (진단) → AI 실행 가능 제안 생성:
    [A1] 제목 최적화 → 3개 대안 제목 + 근거
    [A2] 설명문 최적화 → 리뷰 테마 + 키워드 반영 재작성
    사진 재배열 제안 → 커버 사진 변경, 순서 최적화
    편의시설 체크리스트 → 우선순위별 추가 목록
    [P2][P3] 가격 캘린더 → 향후 90일 날짜별 추천 가격
  표시: /portal/content, /portal/pricing
  주기: 월 1회 자동 + 수동 요청 시

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  STAGE 4: 성과 추적 (Performance Tracking)
  ────────────────────────────────────────
  최적화 전/후 효과 측정:
    [R4] 검색 순위 변화 추적
    [D3] 점유율/수익 트렌드
    기존 GA4/Instagram 분석 (이미 구현됨)
    ★ StayTrend 고유: 국적별 채널 ROI
  저장: properties/{id}/tracking/{category}
  주기: 월 1회 자동 집계
  표시: /portal/analytics + /portal/results

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 2.2 경쟁사 대비 파이프라인 차별점

```
경쟁사들은 파이프라인의 "일부"만 제공:

  RankBreeze  ─── Stage 1 (시장 데이터) + Stage 2 일부 (진단)
  AutoRank    ─── Stage 3 (AI 제안) 에만 집중
  AirDNA      ─── Stage 1 (시장 데이터) + Stage 4 일부 (트렌드)
  PriceLabs   ─── Stage 1 (가격만) + Stage 3 (가격 제안만)

  StayTrend   ─── Stage 0~4 전체 통합
               +  국적별 마케팅 채널 최적화 (고유)
               +  SNS 성과 연동 (고유)
               +  올인원 실행 포털 (고유)
```

### 2.3 구현 우선순위

```
Phase 1 (기반) ──────────────────────────────
  ① Stage 0: 비교군 엔진 (모든 것의 기반)
  ② Stage 1: 편의시설 분석 + 가격 벤치마킹
  → 이것만으로도 "내 숙소 vs 경쟁자" 비교 가능

Phase 2 (진단) ──────────────────────────────
  ③ Stage 2: 리뷰 분석 + 콘텐츠 진단
  ④ Stage 2: 종합 건강 점수 (Health Score)
  → 호스트가 "어디를 개선해야 하는지" 한눈에 파악

Phase 3 (AI 제안) ───────────────────────────
  ⑤ Stage 3: AI 제목/설명 최적화
  ⑥ Stage 3: 가격 캘린더 강화 (비교군 기반)
  → "무엇을 바꿔야 하는지" 구체적 액션 제공

Phase 4 (추적) ──────────────────────────────
  ⑦ Stage 4: 순위/점유율 추적
  ⑧ Stage 4: 최적화 효과 리포트
  → "바꾼 후 효과가 있었는지" 증명
```

## 3. Phase별 기술 스택 & 비용

### 3.1 현재 보유 스택

```
Runtime       Next.js 16 + React 18 + TypeScript
DB            Firestore (Firebase Admin 12)
Scraping      Cheerio 1.2
AI/LLM        OpenAI (gpt-4-turbo-preview)
Auth          Firebase Auth
Visualization Recharts 3.7
Deployment    Vercel (Cron 7개 설정됨)
Integrations  Instagram Graph API, GA4 API, Google OAuth
```

### 3.2 Phase 1 — 비교군 엔진 + 시장 벤치마킹

| 영역 | 선택지 | 추천 | 이유 |
|------|--------|------|------|
| **스크래핑** | Cheerio (기존) / Puppeteer / Playwright / Crawlee | **Crawlee + Playwright** | Airbnb는 JS 렌더링 필요. Crawlee는 자동 재시도/큐/프록시 관리 내장 |
| **프록시** | Bright Data / Oxylabs / ScraperAPI | **ScraperAPI** | 저렴하고 시작 쉬움. Airbnb 대량 스크래핑 시 IP 차단 방지 필수 |
| **비동기 작업** | Vercel Cron (기존) / BullMQ / Inngest / Trigger.dev | **Inngest** | Vercel 함수 300초 제한 우회. 이벤트 기반 작업 체인. Serverless 호환 |
| **데이터 저장** | Firestore (기존) / PostgreSQL | **Firestore (유지)** | 비교군 50~100개 수준에서 Firestore 서브컬렉션 충분 |
| **데이터 처리** | Node.js / Python | **Node.js 직접** | 편의시설 빈도, 가격 백분위 등 단순 집계. 별도 언어 불필요 |

**패키지 추가**:
```
npm install crawlee playwright inngest
# 또는 가벼운 시작:
npm install puppeteer-core @sparticuz/chromium  (Vercel serverless 호환)
```

**비용** (월간, 숙소 100개 기준):

| 항목 | 비용 |
|------|------|
| ScraperAPI (250K 크레딧) | ~$49 |
| Vercel Pro (긴 실행시간) | $20 |
| Firestore (읽기/쓰기) | ~$5~10 |
| **Phase 1 합계** | **~$75~80/월** |

### 3.3 Phase 2 — 리스팅 진단 + 건강 점수

| 영역 | 추천 | 이유 |
|------|------|------|
| **리뷰 감성 분석** | **OpenAI GPT-4o-mini** | 이미 연동됨. 저렴하면서 감성/토픽 분류에 충분 |
| **토픽 추출** | **OpenAI Structured Output** | JSON 스키마로 토픽+감성+점수 한번에 추출 |
| **사진 분석** | **OpenAI GPT-4o (Vision)** | 공간 유형 분류 + 품질 평가 |
| **이미지 품질** | **Sharp** + GPT-4o Vision | Sharp로 해상도/밝기, Vision으로 구도/내용 |
| **점수 산출** | **Node.js 직접** | 6개 카테고리 가중 평균 |

**AI 모델 사용**:
```
리뷰 분석 (GPT-4o-mini, Structured Output):
  Input:  리뷰 텍스트 10~50개 배치
  Output: { sentiment, score, topics[], topPraises[], topComplaints[] }

사진 분석 (GPT-4o Vision):
  Input:  사진 URL
  Output: { spaceType, qualityScore, brightness, composition, suggestion }
```

**패키지 추가**:
```
npm install sharp
# OpenAI는 이미 설치됨 — 모델만 gpt-4o-mini / gpt-4o 사용
```

**비용** (월간 누적):

#### Cheerio → Crawlee + Playwright 전환 이유

```
Cheerio (현재):
  → HTML 문자열을 파싱
  → 문제: Airbnb 상세 페이지의 편의시설, 설명, 리뷰는
    React/JS로 렌더링됨. 정적 HTML에 없음
  → 한계: "더 보기" 클릭, 스크롤, 캘린더 탐색 불가

Crawlee + Playwright (전환):
  → 실제 브라우저 실행 → JS 렌더링 완료 후 데이터 추출
  → "더 보기" 클릭, 스크롤, 캘린더 탐색 가능
  → Crawlee 내장 기능:
    - 자동 재시도 (실패 시 3회)
    - 요청 큐 (URL 목록 순차 처리)
    - 프록시 로테이션 (IP 차단 방지)
    - 동시 실행 제어 (concurrency: 5)
    - 자동 쿠키/세션 관리
```

### 4.3 Step B — 비교군 탐색

#### 검색 전략

```
Airbnb 검색 URL 구조:
  https://www.airbnb.com/s/homes
    ?lat=35.1595                ← 자사 숙소 좌표
    &lng=129.1603
    &zoom=14                    ← 약 3km 반경
    &room_types[]=Entire+home   ← 자사와 동일한 유형
    &min_bedrooms=1             ← 자사 ±1 범위
    &max_bedrooms=3
    &adults=2                   ← 기본 게스트 수
    &checkin=2026-04-15         ← 향후 날짜
    &checkout=2026-04-17

검색 반경 전략 (동적 확장):
  1차: zoom=15 (약 1.5km) → 결과 확인
  2차: zoom=14 (약 3km)   → 부족하면 확장
  3차: zoom=13 (약 6km)   → 최대 확장

  목표: 50~100개 유사 리스팅 수집
```

#### 비교군 필터링 로직

```typescript
interface CompSetCriteria {
  centerLat: number
  centerLng: number
  propertyType: string       // "Entire home"
  roomType: string           // "Entire place"
  accommodates: number       // 자사 수용 인원
  bedrooms: number           // 자사 침실 수
  targetSize: number         // 기본: 75
}

// 필터링 순서:
// 1. 위치 필터: 반경 내 리스팅만
// 2. 유형 필터: 동일 propertyType + roomType
// 3. 규모 필터: accommodates ±2, bedrooms ±1
// 4. 활성 필터: 리뷰 1개 이상, 최근 6개월 내 리뷰
// 5. 거리순 정렬: 가까운 순서로 targetSize개 선택

// 거리 계산: Haversine formula → km 단위
```

### 4.4 Step C — 비교군 상세 스크래핑 (Inngest 작업 체인)

#### 왜 Inngest가 필요한가

```
Vercel Serverless 함수 제한:
  - Pro: 최대 300초 (5분) 실행
  - 리스팅 1개 상세 스크래핑: ~5~10초
  - 75개 리스팅: ~6~12분 → Vercel 제한 초과

Inngest 해결 방식:
  - "이벤트" 기반으로 작업을 잘게 쪼갬
  - 각 단계가 독립적으로 실행 (step function)
  - 자동 재시도, 에러 핸들링, 동시성 제어
```

#### 작업 체인 설계

```
이벤트: "property/onboarding.completed"
  │
  ▼
함수 1: scrape-own-listing
  │  자사 리스팅 상세 스크래핑
  │  → Firestore 저장
  │  → 이벤트: "listing/scrape.completed"
  │
  ▼
함수 2: build-comp-set
  │  Step 1: Airbnb 검색 (반경 1.5km)
  │  Step 2: 부족하면 반경 확장 (3km → 6km)
  │  Step 3: 필터링 → 75개 선정
  │  Step 4: Firestore에 comp-set 목록 저장
  │  → 이벤트: "comp-set/list.ready"
  │
  ▼
함수 3: scrape-comp-details
  │  Step 1~75: 각 리스팅 상세 스크래핑 (step.run 반복)
  │  → 각 step은 독립 실행 (5~10초)
  │  → 실패 시 해당 step만 3회 재시도
  │  → 요청 간 2초 딜레이 (차단 방지)
  │  → 각 결과를 Firestore에 즉시 저장
  │  → 이벤트: "comp-set/scrape.completed"
  │
  ▼
함수 4: analyze-market-intel
  │  Step 1: 편의시설 분석
  │  Step 2: 가격 벤치마킹
  │  Step 3: 스코어카드 생성
  │  → Firestore 저장
  │  → 이벤트: "market-intel/analysis.completed"
  │
  ▼
포털에 결과 표시 준비 완료

예상 소요 시간:
  함수 1: ~10초
  함수 2: ~30~60초
  함수 3: ~5~12분 (75개 × 5~10초, 동시 5개)
  함수 4: ~10~30초
  전체: ~7~15분 (비동기, 호스트는 기다릴 필요 없음)
```

```typescript
    id: "scrape-comp-details",
    concurrency: { limit: 3 },
    retries: 3
  },
  { event: "comp-set/list.ready" },

  async ({ event, step }) => {
    const { propertyId, listingIds } = event.data

    let scraped = 0
    let failed = 0

    for (const listingId of listingIds) {
      await step.run(`scrape-${listingId}`, async () => {
        const data = await scrapeListingDetail(listingId)
        await saveToFirestore(propertyId, listingId, data)
        scraped++
      })

      await step.sleep("wait", "2s")
    }

    await step.sendEvent("notify-complete", {
      name: "comp-set/scrape.completed",
      data: { propertyId, scrapedCount: scraped }
    })
  }
)
```

### 4.5 Step D-1 — 편의시설 격차 분석

#### 분석 로직

```typescript
interface AmenityAnalysis {

    topFrequency: number      // 상위 성과자 보유율

  }[]

```

#### 영향도 점수 계산

```
impactScore = (topPerformerFrequency - allFrequency) × categoryMultiplier

  예시:
  "전기차 충전기": 전체 5%, 상위 25% → gap=20 → 영향도 높음
  "Wi-Fi":       전체 98%, 상위 100% → gap=2  → 기본 필수

  보정 카테고리:
  - 전체 80%+ = "필수 편의시설" (없으면 감점)     → ×3
  - 전체 <10% + 상위 20%+ = "희귀 고효과"        → ×2.5
  - 전체 40~70% = "차별화 기회"                   → ×2
  - 기타                                          → ×1

  난이도 분류:
  - easy:   체크만 하면 됨 (이미 있지만 미표시)
  - medium: 소액 투자로 추가 가능 (커피머신, 넷플릭스 등)
  - hard:   큰 투자 필요 (수영장, 온수욕조 등)
```

#### 출력 예시

```

    주방            비교군 85%  상위 93%   난이도: easy (표시)

    넷플릭스        비교군 41%  상위 67%   난이도: easy

  희귀 고효과 (소수만 있지만 효과 큼):
    전기차 충전기    비교군 5%   상위 20%   난이도: medium
    빔프로젝터      비교군 8%   상위 27%   난이도: medium
```
